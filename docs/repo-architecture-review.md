# ZenaManage Repo Architecture Review

## 1. Executive Summary
- ZenaManage currently implements a broad project-operations platform spanning project/task delivery, RFIs, submittals, change requests, QC inspections, documents, notifications, contracts/payments, and a newer work-template/work-instance deliverable flow. Evidence: active `api/zena` groups for projects, tasks, RFIs, submittals, change requests, inspections, documents, notifications, work templates, deliverable templates, and work instances in `routes/api_zena.php:213-377`; contract/payment routes under `api/v1` in `routes/api.php:821-852`; corresponding models in `app/Models` including `WorkInstance`, `Rfi`, `Submittal`, `ChangeRequest`, `QcInspection`, `Contract`, and `ContractPayment`.
- The repo has two active API surfaces in parallel, but they do not carry equal ownership meaning. For Projects specifically, `/api/zena/projects` is the canonical business API owned by `App\Http\Controllers\Api\ProjectController`, `App\Services\ProjectService`, and `App\Models\Project`, while `/api/v1/projects` remains a mounted compatibility runtime in `Src\CoreProject\Controllers\ProjectController`. Parsing the latest review pack route export at `_repo_review_pack_20260302_130819/20_routes.json` yields 330 `api/v1` routes and 119 `api/zena` routes; the grouped module map in `_repo_review_pack_20260302_130819/21_module_map.md:988-1363` and `_repo_review_pack_20260302_130819/21_module_map.md:1442-1631` is consistent with those prefixes.
- Tenant isolation and RBAC are treated as invariants in both documentation and active middleware. The SSOT rules explicitly prohibit weakening either control in `docs/agent-ssot-rules.md:27-30`, `TenantIsolationMiddleware` enforces `X-Tenant-ID` and rejects mismatches in `app/Http/Middleware/TenantIsolationMiddleware.php:26-103`, `RoleBasedAccessControlMiddleware` enforces permission checks in `app/Http/Middleware/RoleBasedAccessControlMiddleware.php:28-172`, and route-list output shows `Authenticate -> TenantIsolationMiddleware -> RoleBasedAccessControlMiddleware` stacks across `api/zena` and `api/v1`.
- The most coherent current business subsystem is the work-template/work-instance/deliverable export pipeline. It has dedicated MVP schema in `database/migrations/2026_02_27_120000_create_work_templates_mvp_tables.php:13-191` and `database/migrations/2026_02_27_130000_create_work_instance_step_attachments_table.php:9-33`, tenant-scoped models such as `app/Models/WorkInstance.php:12-41`, API controllers in `app/Http/Controllers/Api/WorkTemplateController.php:21-320`, `app/Http/Controllers/Api/WorkInstanceController.php:28-320`, `app/Http/Controllers/Api/DeliverableTemplateController.php:19-320`, export services in `app/Services/DeliverablePdfExportService.php:10-260` and `app/Services/WorkInstanceExportBundleService.php:16-262`, and dedicated feature tests under `tests/Feature/Api/WorkTemplateMvpApiTest.php`, `tests/Feature/Api/DeliverableTemplateMvpApiTest.php`, `tests/Feature/Api/WorkInstanceDeliverableExportApiTest.php`, and `tests/Feature/Api/WorkInstanceExportBundleApiTest.php`.
- Inspection/quality, RFI, submittal, and change-request modules are implemented and tenant-aware, but they are less cleanly bounded than the newer work-instance flow. Evidence: route groups in `routes/api_zena.php:259-307`; controllers `app/Http/Controllers/Api/RfiController.php:16-280`, `app/Http/Controllers/Api/SubmittalController.php:16-280`, `app/Http/Controllers/Api/ChangeRequestController.php:17-300`, and `app/Http/Controllers/Api/InspectionController.php:16-260`; policies in `app/Policies/RfiPolicy.php`, `app/Policies/QcInspectionPolicy.php`, and `app/Policies/ChangeRequestPolicy.php`; migrations `database/migrations/2025_09_20_133629_create_rfis_table.php:16-71`, `database/migrations/2025_09_20_141930_create_qc_plans_table.php:16-39`, `database/migrations/2025_09_20_142005_create_qc_inspections_table.php:16-42`, and `database/migrations/2025_09_17_162350_create_change_requests_table.php:16-67`.
- Role-specific operational support exists, especially for PM, designer, and site engineer personas, but dashboard ownership is split between `api/zena`, `api/v1`, and even web-mounted API-style routes. Evidence: role dashboard routes in `routes/api_zena.php:185-210`; latest-pack keyword hits for `DashboardRoleBasedService` role/quality widgets in `_repo_review_pack_20260302_130819/30_keyword_hits.txt:7589-7609`; additional dashboard routes under `routes/api.php:675-900`; and web-mounted API-style routes in `routes/web.php:75-190`.
- Export/reporting depth is stronger than average for the current codebase: HTML, deploy-safe PDF, and ZIP bundle contracts are documented in `docs/deploy-runbook.md:45-141`; PDF export depends on Node and Playwright in `app/Services/DeliverablePdfExportService.php:28-247`; ZIP bundles include `manifest.json`, `deliverable.html`, optional `deliverable.pdf`, and attachments in `app/Services/WorkInstanceExportBundleService.php:45-158`; and route-list confirms `POST api/zena/work-instances/{id}/export` and `POST api/zena/work-instances/{id}/export-bundle`.
- Observability is present but centered on audit logging rather than deeper tracing. `ZenaAuditLogger` persists sanitized audit records in `app/Services/ZenaAuditLogger.php:32-171`, and the newer controllers call it on create/update/export operations, for example `app/Http/Controllers/Api/WorkTemplateController.php:125-136`, `app/Http/Controllers/Api/DeliverableTemplateController.php:83-94`, `app/Http/Controllers/Api/InspectionController.php:161-171`, and `app/Http/Controllers/Api/WorkInstanceController.php:150-161`.
- The largest technical debt themes are route duplication and mixed module ownership. Evidence: canonical `api/v1` is defined in `routes/README.md:3-8`, but business-critical modules also live in `routes/api_zena.php:213-377`; `routes/web.php:75-190` exposes API-like routes under `/api/v1/...`; and code is split across both `app/*` and `src/*` (`composer.json:40-47`).
- Evidence gaps remain and are called out instead of guessed. The latest pack is `_repo_review_pack_20260302_130819`, and its manifest explicitly includes `20_routes.json`, `21_module_map.md`, and `30_keyword_hits.txt` in `_repo_review_pack_20260302_130819/README_FOR_CODEX.md:3-11`. Submittal table provenance is still partially legacy: `zena_submittals` is created in `database/migrations/2025_09_14_110000_create_zena_system_tables.php:111-133` and tenant-patched in `database/migrations/2026_02_05_000000_add_tenant_id_to_zena_submittals_table.php:11-31`, but no `Schema::create('submittals')` migration was found in `database/migrations`, so the current physical `submittals` table creation remains `UNKNOWN`.

## 4. System Design Assessment

### Strengths
- Tenant isolation is explicit, centralized, and enforced before downstream module logic. `TenantIsolationMiddleware` resolves tenant context from `X-Tenant-ID` or authenticated user, rejects mismatches, rejects missing tenants, and binds `current_tenant_id` in the container (`app/Http/Middleware/TenantIsolationMiddleware.php:37-103`). The kernel prioritizes `Authenticate`, then `TenantIsolationMiddleware`, then `RoleBasedAccessControlMiddleware` (`app/Http/Kernel.php:80-103`), and route-list output for both `api/zena` and `api/v1` shows that ordering on active business routes.
- RBAC is route-attached rather than hidden in controller branches for most current surfaces. `RoleBasedAccessControlMiddleware` checks named permissions via persisted permission codes (`app/Http/Middleware/RoleBasedAccessControlMiddleware.php:133-172`), and route-list evidence shows per-endpoint permissions such as `template.publish`, `work.export`, `inspection.complete`, `rfi.respond`, and `contract.payment.create`.
- The export pipeline is unusually complete for the newer work-instance module. HTML rendering is tied to deliverable template versions; PDF generation degrades safely when Playwright/Chromium is missing (`app/Services/DeliverablePdfExportService.php:41-47`, `173-197`); ZIP generation adds manifest plus attachments and optional PDF (`app/Services/WorkInstanceExportBundleService.php:45-158`); and the deploy runbook defines smoke checks and expected contracts for HTML/PDF/ZIP (`docs/deploy-runbook.md:50-141`).
- The newer work-template module has strong schema coherence and versioning semantics. Draft and published versions are separated in controller logic (`app/Http/Controllers/Api/WorkTemplateController.php:99-123`, `221-252`), immutable published versions are supported in schema (`database/migrations/2026_02_27_120000_create_work_templates_mvp_tables.php:13-35`), and template package import/export has an explicit schema version contract (`app/Services/WorkTemplatePackageService.php:15-22`).
- Audit logging is consistently used in the newer and operationally sensitive controllers. `ZenaAuditLogger` writes sanitized route/action metadata to `audit_logs` (`app/Services/ZenaAuditLogger.php:32-171`), and controllers call it for template creation/publish, deliverable upload/publish, inspection mutations, and work-instance export/update actions (`app/Http/Controllers/Api/WorkTemplateController.php:125-136`, `254-266`; `app/Http/Controllers/Api/DeliverableTemplateController.php:83-94`, `237-249`, `298-310`; `app/Http/Controllers/Api/InspectionController.php:161-171`, `251-260`; `app/Http/Controllers/Api/WorkInstanceController.php:150-161`, `203-216`, `291-302`).
- Testing strategy is documented and includes both invariant enforcement and module-specific API coverage. The testing matrix defines `composer test:fast`, grouped nightly suites, and SSOT lint usage in `docs/engineering/testing-matrix.md:5-120`. The repo contains dedicated invariant suites under `tests/Feature/Zena/*` and route/middleware contract tests under `tests/Feature/RouteMiddleware*`, plus concrete MVP tests for work templates, deliverables, and exports (`tests/Feature/Api/WorkTemplateMvpApiTest.php`, `tests/Feature/Api/DeliverableTemplateMvpApiTest.php`, `tests/Feature/Api/WorkInstanceDeliverableExportApiTest.php`, `tests/Feature/Api/WorkInstanceExportBundleApiTest.php`).
- ULID/string identifiers are used consistently in the newer modules and many core operational tables, which aligns with the stated invariant that ULID remains string-based. Evidence: `app/Models/WorkInstance.php:14-18`, `app/Models/DeliverableTemplateVersion.php:13-18`, `app/Models/Rfi.php:13-16`, `app/Models/QcInspection.php:18-18`; migrations use `ulid(...)` or string ULID comments in `database/migrations/2026_02_27_120000_create_work_templates_mvp_tables.php:13-191`, `database/migrations/2025_09_20_141930_create_qc_plans_table.php:17-25`, `database/migrations/2025_09_20_142005_create_qc_inspections_table.php:17-24`, `database/migrations/2025_09_20_133629_create_rfis_table.php:17-18`, and `database/migrations/2025_09_17_162350_create_change_requests_table.php:17-19`.

### Weaknesses / technical debt
- The route surface is fragmented across `api/v1`, `api/zena`, and web-mounted `/api/v1/...` endpoints. `routes/README.md:5-8` declares `routes/api.php` as canonical for `api/v1`, while business modules also live in `routes/api_zena.php:12-377`, and `routes/web.php:75-190` mounts API-style prefixes like `/api/v1/universal-frame`, `/api/v1/accessibility`, `/api/v1/performance`, and `/api/v1/final-integration` under the web router. This increases discovery cost and weakens clean modular boundaries.
- Module ownership is inconsistent between `app/*` and `src/*`. Composer autoload maps both `App\` and `Src\` namespaces (`composer.json:40-47`), and the repo has domain modules under `src/ChangeRequest`, `src/CoreProject`, `src/DocumentManagement`, `src/WorkTemplate`, and `src/RBAC` while current `api/zena` controllers for the same business areas live under `app/Http/Controllers/Api`. This is visible, for example, in both `app/Http/Controllers/Api/ChangeRequestController.php` and `src/ChangeRequest/Controllers/ChangeRequestController.php`.
- Permission naming is not fully normalized across route surfaces. `api/zena` uses tokens like `change-request.view` and `template.edit_draft` in `routes/api_zena.php:226-306`, while `api/v1` change-request routes from route-list use `change_request.view` and older `src/*` ownership. This is a real contract split that complicates policy and permission governance.
- Review-pack route ownership evidence is still prefix-level rather than bounded-context level. `_repo_review_pack_20260302_130819/21_module_map.md:1-3` explicitly describes the file as an evidence-derived "module-ish map" by URI prefix, so exact route-to-module ownership for mixed `api/v1` and `src/*` surfaces remains `UNKNOWN` without direct source inspection.
- Some modules are present in models/policies without a clean, obvious current API boundary. NCRs are a concrete example: the `Ncr` model and migration exist, and `QcInspection` has an `ncrs()` relation, but this review did not confirm an active NCR API surface. Similar split ownership exists between document code in `src/DocumentManagement/*` and the active `SimpleDocumentController`.
- Web routes still carry API-like behavior and security semantics that belong in API route files. `routes/web.php:75-190` defines `/api/v1/universal-frame/*`, `/api/v1/accessibility/*`, `/api/v1/performance/*`, and `/api/v1/final-integration/*` with auth, tenant, and RBAC middleware. That is a real maintenance risk because route intent is no longer aligned to file ownership.
- Observability is mostly audit-log centric. Audit logs are valuable, but this review did not confirm deeper request tracing, queue instrumentation, or export-specific metrics beyond logs and smoke tests. Queue requirements are documented in `docs/deploy-runbook.md:8-18`, but module-level telemetry for long-running export/report workflows is not evident from the inspected code, so deeper metrics/tracing appear limited.
- Test coverage is uneven by module maturity. The work-template/export subsystem has targeted feature and unit tests, while this review did not identify equivalent focused suites for RFIs, submittals, inspections, or contracts/payments. The testing matrix is strong at framework level (`docs/engineering/testing-matrix.md:60-125`), but module-level depth is concentrated in the newer MVP pipeline.
